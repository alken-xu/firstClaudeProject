## 1. 全体ストーリーライン（3セッション通し）

### Phase 1: 要件定義セッション
1. **プロンプト設計の相談** → フェーズ別プロンプトの提示
2. **要件定義の自動生成** → 参考サイトを参照した上でMarkdown形式の要件定義書を作成
3. **GitHub連携** → `git init` → リポジトリ作成 → push まで自動実行
4. **技術スタックの議論** → ユーザーの技術レベルに合わせて最適な選択肢を提示・決定
5. **要件定義書の更新** → 技術スタック確定後に自動更新・再push
6. **セッション管理設計** → CLAUDE.md + session-*.md による2セッション分離運用を設計・実装
7. **メモリ保存** → 次回セッションへの引き継ぎ情報を永続化

### Phase 2: 開発セッション
8. **プロジェクト初期構築** → フロントエンド・バックエンドを要件定義書通りにスキャフォールド
9. **DB設計・シードデータ** → テーブル定義・サンプルデータを自動生成
10. **画像収集** → Unsplashから旅館テーマに沿った画像を検索・一括ダウンロード
11. **地図埋め込み** → OpenStreetMapをAPIキー不要で埋め込み
12. **メール送信機能** → Gmail SMTPによる予約・問い合わせ通知メールを実装
13. **バリデーション強化** → 日本の電話番号形式（携帯/固定/IP/フリーダイヤル）に対応
14. **予約確認フロー** → 4ステップ化（入力→選択→お客様情報→**確認画面**→完了）
15. **GitHub push** → 全変更を都度コミット・プッシュ

## 2. ClaudeCodeの機能デモポイント（PPT向け）

> テストセッションは **2-12 〜 2-13** に追記。

### ▼ 要件定義セッション

#### 2-1. Web情報の参照と活用
- **実演内容**: 参考サイト（hatagoya-maruichi.com）を参照し、旅館予約サイトの構成を把握した上で要件定義を生成
- **ポイント**: URLを渡すだけで「サイト構成を読み取り → 要件に落とし込む」まで自動
- **補足**: token消費コントロールの観点からトップページのみ参照に絞った（ユーザーが途中で指示）

#### 2-2. ドキュメント自動生成（要件定義書）
- **実演内容**: ページ構成・データモデル・API設計・技術スタック・開発優先順位を含む要件定義書（Markdown）を一度に生成
- **ポイント**: 観点を指定したプロンプトで質を担保。生成後もユーザーとの対話で内容を洗練
- **生成ファイル**: `requirements.md`（v1.1）

#### 2-3. GitHub操作の自動化
- **実演内容**: `git init` → `gh repo create` → `git push` をすべて自動実行
- **ポイント**: GitHub CLIのパスを自動探索して解決（環境依存の問題も自力で解消）
- **補足**: 事前に `gh auth login` 済みであれば、リポジトリ作成からpushまでノーコード

#### 2-4. 技術スタック選定の支援
- **実演内容**: ユーザーのスキル（JS/Node.js経験あり・React未経験）を踏まえて選択肢を比較提示
- **ポイント**: 最新技術を押し付けず、ユーザーが自力でコードを読める範囲を優先して提案
- **決定内容**: React + Vite + Tailwind CSS / Express + SQLite（better-sqlite3）

#### 2-5. セッション設計とメモリ機能
- **実演内容**: 要件定義と開発を分離した2セッション運用の仕組みを設計・実装
- **ポイント**: ClaudeCodeの記憶の限界をファイルベース（CLAUDE.md / session-*.md）で補う設計
- **メモリ**: プロジェクト情報・ユーザープロフィールを `memory/` に永続保存し次回セッションに引き継ぎ

### ▼ 開発セッション

#### 2-6. フルスタック初期構築（ゼロから一気に）
- **実演内容**: `npm create vite` → Tailwind設定 → Express/SQLiteセットアップ → 全ページ・全API実装を一括実行
- **ポイント**: 要件定義書を読み込み、仕様通りのフォルダ構成・ファイルを自動生成
- **成果**: frontend（10ページ）+ backend（5 APIルート）+ DBシードデータを1セッションで完成
- **実際の問い合わせ**: 「CLAUDE.mdをベースに、プロジェクト初期構築してください。」の一言

#### 2-7. 外部ライブラリ・環境トラブルの自律解決
- **実演内容**: `better-sqlite3` がNode.js v22向けビルド済みバイナリなしでインストール失敗
- **解決過程**: エラーログを読んで原因特定 → npmバージョン指定（v9.6.0→v12.8.0）で自己解決
- **ポイント**: ユーザーが何もしなくても、エラー → 調査 → 解決を自律的に完結

#### 2-8. 画像収集・外部サービス連携
- **実演内容**: 「写真準備中」プレースホルダーを実際の画像に差し替え
- **工程**:
  1. WebSearchでUnsplashのPhoto IDを検索（テーマ別に並行検索）
  2. ダウンロードURLを一覧で提示（ユーザーが内容を確認）
  3. 20枚を並行ダウンロード → `public/images/` に配置
  4. フロントエンドのプレースホルダー部分を `<img>` タグに自動差し替え
  5. ヒーロー部分は5秒自動切替スライダーに改修
- **ポイント**: 検索・ダウンロード・コード修正・確認まで一気通貫

#### 2-9. 地図埋め込み（APIキー不要）
- **実演内容**: アクセスページの「地図（準備中）」を OpenStreetMap iframe に差し替え
- **ポイント**: 「APIキー不要・無料」という制約を自ら判断して最適解を選択
- **追加対応**: ズームレベルを「旅館周辺」→「関東全域」に変更（1行修正）

#### 2-10. メール送信機能の実装
- **実演内容**: 予約完了・お問い合わせ送信時に旅館オーナーへHTMLメール通知
- **ユーザーとのやり取り（デモ見せ場）**:
  - 「送信元はユーザーが入力したアドレスにしたい」→ SMTPの制約を説明し Reply-To で代替提案
  - アプリパスワードの取得で詰まる → 直接URLを案内して解決（myaccount.google.com/apppasswords）
- **技術実装**: nodemailer + Gmail SMTP / 非同期送信（レスポンス速度に影響なし）
- **実際の問い合わせ**: 「予約した後、予約情報をxuguoqi00@gmail.comに送信してくれますか。」

#### 2-11. バリデーション + UXフロー改善
- **実演内容**: 2つの改善を1リクエストで同時対応
  1. 電話番号バリデーション（携帯/固定/IP電話/フリーダイヤル対応の正規表現）
  2. 予約確認画面の追加（入力内容を確認してから確定）
- **ポイント**: スクリーンショット（画像）を渡して「この画面を改善して」という自然な指示で動作
- **実際の問い合わせ**: 「[スクリーンショット] の予約入力画面で電話番号を日本における有効な電話番号へのバリエーションを追加、予約入力した後、一度入力情報をユーザーに表示させて確認を取れた後、確定させるように修正してください。」

### ▼ テストセッション

#### 2-12. 自動単体テスト基盤の構築（Playwright + Excel レポート）
- **実演内容**: Playwright を使った自動テストスクリプトを作成し、テスト結果を Excel に出力
- **ユーザーの一言**: 「私がインストールしたplaywrightを利用し、自動単体テストを実施したいです。あなたが単体テストのEXCELファイルを作成し、ここの単体テストも実施してもらえますか。」
- **Claudeが自律実施したこと**:
  1. プロジェクトのページ構成・フォーム仕様を読み込み → テストケース16件を設計
  2. `tests/package.json` 作成 → `npm install`（playwright + exceljs）
  3. `npx playwright install chromium` でブラウザを自動インストール
  4. `run-tests.js`（約400行）を生成 — テストロジック + Excel生成を一ファイルに統合
  5. サーバー起動確認（curl）→ テスト実行 → 全16件 PASS → Excel出力まで一気通貫
- **技術ポイント**:
  - React Hook Form のバリデーション動作（`p.text-red-500` セレクター等）を読んで正確なセレクターを設計
  - 予約フロー Step1〜4 を完全自動化（日付入力 → 客室選択 → 顧客情報 → 確認画面）
  - 各テストケースを独立実行（前のテスト状態に依存しない設計）
- **Excel 出力仕様**:
  - Sheet 1「テストケース一覧」: 全16件の ID・手順・期待結果・実施結果を色分け一覧化
  - Sheet TC-001〜TC-016（各16シート）: テスト情報 + スクリーンショット（エビデンス）
- **結果**: 16件 / 16件 PASS（初回実行）、合計実行時間 約40秒

## 4. セッション運用設計（2セッション分離）

```
firstClaudeProject/
├── CLAUDE.md                ← 共通情報（両セッションが参照）
├── session-requirements.md  ← 要件定義セッション専用ルール
├── session-dev.md           ← 開発セッション専用ルール
└── requirements.md          ← 要件定義書（要件定義セッションのみ編集可）
```

### 分離の目的
- 要件定義の変更と実装の変更を混在させない
- ミスが発生しても2つのセッションが直接連動して修正しない
- 修正はユーザーが各セッションで明示的に承認した上で行う

### セッション開始方法
```
# 要件定義セッション
「session-requirements.mdを読んで、その指示に従って作業してください」

# 開発セッション
「session-dev.mdを読んで、その指示に従って作業してください」
```

## 7. 補足メモ（PPT作成時の注意点）

- **所要時間の目安**: 要件定義セッション約1時間 + 開発セッション約2〜3時間 + テストセッション約30分（会話ベースで実施）
- **コスト**: ClaudeCode Pro プランで利用（月額定額）
- **Claudeが苦手だったこと**:
  - Unsplashの直接fetch（403エラー） → WebSearchで代替
  - 画像の日本語的テーマとの厳密な一致（風景写真で代替）
- **Claudeが得意だったこと**:
  - エラーの自律解決
  - 技術的制約の説明と代替案提示（SMTP制約 → Reply-To方式）
  - ユーザーの安全確認への対応（URL確認してから実行）
  - 複数ファイルの一貫した修正
  - コードを読んでテストケースを自動設計（セレクター・フロー・バリデーション）
  - テスト結果をビジネス用途向けの Excel レポートとして出力

### テストセッションのデモ見せ場（追加）
- **「一言」の威力を示す最適例**: テスト設計・実装・実行・Excel出力 を1リクエストで完結
- **Excel レポートをそのまま納品物として使える**: スクリーンショット付きエビデンスシートは業務で実際に使えるクオリティ
- **Playwright MCP の活用**: `@playwright/mcp` を `.mcp.json` に設定するだけでブラウザ操作が可能になる点も説明ポイント
